# Meeting Summarizer

여러 개의 회의록 파일을 입력받아 회의 전반에 흩어진 사건들을 추출하고, 유사 사건을 병합한 뒤, 사건별 **발생 → 논의 → 조치 → 결과** 흐름을 정리해 Markdown/JSON 리포트로 출력하는 회의록 사건 분석 프로젝트입니다.

이 프로젝트의 1차 목표는 웹서비스가 아니라, 로컬 환경에서 회의록 파일을 처리하고 분석 결과를 파일 형태로 안정적으로 저장하는 것입니다.

---

## 1. 프로젝트 목표

1차 MVP의 목표는 다음과 같습니다.

> PDF/DOCX/TXT 회의록 여러 개를 입력받아 사건 후보를 추출하고, 유사 사건을 병합한 뒤, 사건별 발생-논의-조치-결과 리포트를 Markdown/JSON으로 출력한다.

즉, 단순 회의록 요약기가 아니라 **회의록 전체에서 사건 단위의 흐름을 추적하는 분석 도구**입니다.

예를 들어 여러 회의록에 다음과 같이 흩어진 내용이 있을 수 있습니다.

- 1차 회의록: 특정 문제가 처음 언급됨
- 2차 회의록: 해당 문제에 대한 조치 방향이 논의됨
- 3차 회의록: 담당자와 처리 일정이 정해짐
- 4차 회의록: 결과 또는 남은 이슈가 보고됨

이 프로젝트는 이런 흩어진 언급들을 하나의 사건 케이스로 묶고, 시간 흐름에 따라 정리합니다.

## 2. 주요 기능

### 2.1 회의록 파일 입력

1차 MVP에서 지원하는 입력 파일 형식은 다음과 같습니다.

- `.txt`
- `.docx`
- `.pdf`

HWP/HWPX 파일은 1차 구현 범위에서 제외하며, 추후 확장 대상으로 둡니다.

### 2.2 파싱 및 세그먼트 분할

입력된 회의록 파일은 코드 기반 파서로 텍스트를 추출합니다.

파싱/전처리 단계에서는 LLM을 사용하지 않습니다.

기본 세그먼트 분할 기준은 다음과 같습니다.

- PDF/DOCX: 2페이지 단위로 분할
- TXT: 페이지 개념이 없으므로 구현자가 합리적인 fallback 기준을 정하고 문서화

회의 안건별 세그먼트 분할은 추후 고도화 대상으로 둡니다.

### 2.3 사건 후보 추출

각 세그먼트에서 LLM을 사용해 사건 후보를 추출합니다.

사건 후보는 최종 사건이 아니라, 특정 회의록 세그먼트에서 발견된 사건 단위의 원시 후보입니다.

### 2.4 사건 후보 임베딩 및 유사 후보군 생성

추출된 사건 후보의 `title`, `summary`, `embedding_text` 등을 기반으로 임베딩을 생성합니다.

임베딩 결과는 FAISS 기반 벡터 저장소에 저장합니다.

유사도 기준은 다음과 같습니다.

- cosine similarity `> 0.8`: 같은 사건 후보군으로 묶음
- cosine similarity `<= 0.8`: 별도 사건으로 처리

### 2.5 LLM 기반 최종 사건 병합

임베딩 유사도 기준으로 묶인 후보군을 LLM 입력으로 전달합니다.

LLM은 여러 개의 `EventCandidate` JSON을 바탕으로 하나의 최종 `EventCase` JSON을 생성합니다.

즉, 사건 병합은 단순 중복 제거가 아니라, 여러 회의록에 흩어진 정보를 종합해 하나의 사건 케이스로 구성하는 단계입니다.

### 2.6 타임라인 정리

최종 사건 케이스별로 발생, 논의, 조치, 결과 흐름을 정리합니다.

날짜 해석은 별도 코드로 정규화하지 않고 LLM에 맡깁니다.

날짜가 애매한 경우에는 무리하게 추정하지 않고 판단을 보류합니다.

초기 타임라인은 필요한 경우 해당 사건이 처음 등장한 회의록 기준으로 정리할 수 있습니다.

### 2.7 최종 리포트 생성

최종 분석 결과는 다음 두 형식으로 출력합니다.

- 내부 처리 결과: JSON
- 사용자용 리포트: Markdown

PDF/HTML 출력은 추후 확장 대상으로 둡니다.

---

## 3. 전체 처리 흐름

```text
회의록 파일들
→ 파싱/전처리
→ 2페이지 단위 세그먼트 생성
→ LLM으로 사건 후보 EventCandidate 추출
→ 후보별 embedding_text 임베딩
→ FAISS에 벡터 저장
→ 유사도 0.8 초과 후보군 grouping
→ 후보군 JSON을 LLM에 입력
→ 최종 EventCase 생성
→ 타임라인 정리
→ JSON 저장
→ Markdown 리포트 생성
```

## 4. LLM 사용 원칙

이 프로젝트는 모든 단계에 LLM을 사용하는 방식이 아닙니다.

| 단계 | LLM 사용 여부 | 설명 |
|---|---:|---|
| 파싱/전처리 | 사용 안 함 | 파일에서 텍스트와 세그먼트를 추출하는 단계는 코드로 처리 |
| 사건 후보 추출 | 사용 | 각 세그먼트에서 사건 후보 JSON 생성 |
| 사건 병합 | 사용 | 임베딩으로 묶인 후보군을 LLM이 하나의 사건 케이스로 병합 |
| 타임라인 정리 | 사용 | 사건별 발생-논의-조치-결과 흐름 정리 |
| 최종 리포트 생성 | 사용 | 정리된 사건 데이터를 기반으로 Markdown 리포트 생성 |

최종 리포트 생성 단계에서는 전체 회의록 원문을 다시 읽지 않고, 정리된 사건 데이터만 사용합니다.

## 5. 실행 환경

1차 구현은 로컬 실행을 기준으로 합니다.

기본 목표 환경은 다음과 같습니다.

- GPU: RTX 5080 16GB
- LLM: Qwen2.5 14B Q4 양자화 모델
- 임베딩 모델: 미확정
- 벡터 저장소: FAISS
- 결과 저장: JSON 파일 + Markdown 파일

모델 경로, 임베딩 모델, 저장 경로 등은 설정 파일이나 환경 변수로 교체 가능하게 설계하는 것을 권장합니다.

## 6. 데이터 스키마

이 프로젝트의 사건 데이터는 두 단계로 나뉩니다.

1. 사건 후보 스키마: `EventCandidate`
2. 최종 사건 스키마: `EventCase`

### 6.1 EventCandidate

`EventCandidate`는 각 세그먼트에서 추출된 사건 후보입니다.

```python
class EventCandidate:
    candidate_id: str
    meeting_id: str
    segment_id: str
    source_file: str

    title: str
    summary: str

    occurred_at: str | None
    actors: list[str]

    problem: str | None
    discussion: str | None
    action: str | None
    result: str | None

    status: str
    evidence_text: str
    keywords: list[str]
    embedding_text: str
```

### 6.2 EventCase

`EventCase`는 여러 사건 후보가 병합된 최종 사건 케이스입니다.

```python
class EventCase:
    case_id: str
    title: str
    summary: str

    candidate_ids: list[str]
    related_meeting_ids: list[str]

    first_occurred_at: str | None
    actors: list[str]

    occurrence: str | None
    discussion: str | None
    actions: list[str]
    result: str | None

    status: str
    remaining_issues: list[str]
    evidence: list[EvidenceSpan]
    timeline: list[TimelineItem]
```

`EvidenceSpan`과 `TimelineItem`의 세부 필드는 구현 과정에서 정하되, 최종 사건 스키마와 호환되도록 관리합니다.

## 7. 출력 리포트 형식

사용자용 Markdown 리포트는 사건별로 다음 구조를 따릅니다.

```markdown
## [사건명]

- 최초 발생:
- 관련 회의록:
- 사건 내용:
- 처리 과정:
- 담당자:
- 최종 결과:
- 현재 상태:
- 남은 이슈:
- 근거:
```

상태값은 기본적으로 다음 중 하나를 사용합니다.

- 해결 완료
- 진행 중
- 미해결
- 확인 필요

---

## 8. 저장 방식

1차 구현에서는 웹서비스화를 고려하지 않습니다.

로컬 파일 기반으로 다음 결과를 저장합니다.

- 파싱 결과 JSON
- 세그먼트 JSON
- 사건 후보 JSON
- 최종 사건 케이스 JSON
- FAISS 벡터 인덱스
- 사용자용 Markdown 리포트

저장 디렉터리 구조는 구현자가 프로젝트 구조와 사용 편의성을 고려해 결정할 수 있습니다.

단, 저장 구조를 결정하거나 변경한 경우 반드시 문서화해야 합니다.

---

## 9. 프롬프트 관리

LLM 프롬프트는 코드 내부에 직접 작성하지 않고 별도 파일로 관리합니다.

예상 구조는 다음과 같습니다.

```text
prompts/
  event_extraction.md
  event_case_merge.md
  timeline_builder.md
  final_report.md
```

이 방식은 프롬프트 수정과 버전 관리를 쉽게 하기 위한 것입니다.

## 13. README.md와 AGENTS.md의 역할

이 문서인 `README.md`는 프로젝트를 처음 보는 사람이 프로젝트의 목적, 기능, 처리 흐름, 입력/출력 구조를 이해하기 위한 문서입니다.

구현 원칙, 에이전트별 책임, 코딩 규칙, 금지사항, 테스트 보고 규칙 등은 `AGENTS.md`에서 관리합니다.

- `README.md`: 사람에게 프로젝트를 설명하는 문서
- `AGENTS.md`: Codex/에이전트에게 구현 원칙을 지시하는 문서
